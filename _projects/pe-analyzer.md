## 프로젝트 배경

Windows PE 파일 분석 과정에서는 분석 기준이 분석가의 경험에 의존하는 경우가 많아, 분석 누락, 오탐(false positive), 분석 시간 편차와 같은 문제가 발생합니다. 이러한 비효율은 초기 트리아지 단계에서의 판단 정확도를 저하시킬 뿐만 아니라, 악성코드 대응 속도와 분석 결과의 일관성에도 영향을 미칩니다.
따라서 분석 과정의 표준화와 효율성 향상을 위해 핵심 분석 기능을 통합한 도구의 필요성이 대두되었습니다.

## 주요 기능

### 1. PE 헤더 파싱

DOS Header, NT Header (File Header + Optional Header), Section Header 정보를 `QTableWidget`으로 시각화하고, 각 필드의 원시 값과 의미를 함께 제공합니다. 이를 통해 비정상적인 헤더 값이나 구조적 이상 여부를 직관적으로 식별할 수 있으며, 악성코드 의심 포인트를 빠르게 탐지할 수 있도록 지원합니다.

| 영역 | 주요 필드 |
|------|-----------|
| DOS Header | `e_magic` (MZ 시그니처), `e_lfanew` (NT Header 오프셋) |
| NT Header | Machine, TimeDateStamp, AddressOfEntryPoint, ImageBase, SectionAlignment |
| Section Header | Name, VirtualAddress, SizeOfRawData, Characteristics |

### 2. 패킹 여부 검사

섹션 이름 기반 시그니처를 활용하여 UPX, ASPack, ASProtect, PECompact, Krypton 등 주요 패커를 자동 탐지합니다. 패킹 여부는 코드 분석 가능성을 결정하는 핵심 기준으로, 패킹이 적용된 경우 정상적인 정적 분석이 제한되기 때문에 선행적으로 언패킹 여부를 판단해야 합니다. 본 기능은 패킹 감지 시 즉시 경고를 제공하여, 분석 방향을 빠르게 결정할 수 있도록 지원합니다.

### 3. Entry Point 상세 분석

Entry Point(EP) 기준으로 Section 정보, File Offset, First Bytes, Linker 및 Compiler 정보를 제공하여 실행 흐름의 시작 지점을 분석할 수 있도록 했습니다. EP 영역의 바이트 패턴과 위치 정보를 통해 패킹 또는 난독화 여부를 판단할 수 있으며, 비정상적인 진입 구조를 식별하여 악성코드 의심 여부를 빠르게 파악할 수 있도록 지원합니다.

### 4. 헥스 뷰어 & 에디터

바이너리 데이터를 `오프셋 | 헥스 값 | ASCII` 형식으로 시각화하고, 특정 바이트 패턴 검색 및 오프셋 기반 편집 기능을 제공합니다. 이를 통해 분석 중 식별한 의심 구간을 즉시 수정하고 재확인할 수 있어, 외부 도구로 전환하지 않고도 분석–수정–검증 과정을 연속적으로 수행할 수 있습니다. 또한 PE 형식이 아닌 파일에서도 헥스 뷰어가 동작하도록 예외 처리를 구현하여, 다양한 파일에 대한 일관된 분석 흐름을 유지했습니다.
### 5. PDF 분석 보고서 자동 생성

ReportLab을 활용하여 분석 결과를 PDF로 자동 생성하고, 작성 일시, DOS/NT/Section Header 정보, 상세 분석 내용 등을 포함하여 기록합니다. 이를 통해 분석 결과를 문서 형태로 보존하고 공유할 수 있어, 포렌식 분석 과정에서의 증거 기록 및 재현성을 확보할 수 있습니다.

## 담당 업무

### 일정·리스크 관리

팀장으로서 기능 간 의존 관계를 분석해 병렬 개발 구간을 도출하고, 한정된 일정 내 핵심 기능 구현을 완료했습니다. 개발 과정에서 pefile.PEFormatError가 발생할 경우 프로그램이 종료되어 분석이 중단되는 문제를 확인하고, 이를 전체 시스템 안정성에 영향을 주는 리스크로 판단했습니다.
이에 따라 예외 처리를 최우선 과제로 격상하여 비정상 PE 구조나 손상된 파일에서도 프로그램이 종료되지 않도록 개선했으며, 이를 통해 다양한 바이너리 환경에서도 분석이 지속 가능한 안정성을 확보했습니다.

### UI/UX 설계 (Qt Designer)

<img src="{{ site.baseurl}}/assets/images/UI.png" alt="UI/UX 디자인 설계" width="100%">

초보 분석가도 직관적으로 사용할 수 있는 원클릭 워크플로우를 목표로 Qt Designer에서 `.ui` 파일로 먼저 프로토타이핑 했습니다.

- PEview·HxD·Detect It Easy의 화면 구성을 벤치마킹하여 **파일 선택 → 헤더 분석 → 상세 정보 → 헥스 뷰어 → PDF 보고서** 흐름을 단일 창에 구성
- 상세 정보·헥스 뷰어는 다중 창(QDialog)으로 분리하여 메인 화면 복잡도를 최소화
- 패킹 감지 시 빨간색 경고, 분석 단계별 버튼(Detail / HEX / PDF)을 상단 배치하여 자연스러운 이동 흐름 구성
- 팀원 피드백을 반영하여 레이아웃을 수정하고 최종 디자인 확정

### 헥스 뷰어 개발 (`hex_viewer.py`)

별도 헥스 에디터 없이 바이너리를 확인·수정할 수 있는 뷰어·에디터를 직접 구현했습니다.

- 파일을 바이너리 모드(`rb`)로 읽어 16바이트 단위로 파싱, 출력 불가능한 문자는 `.`으로 치환하여 가독성 확보
- 바이트 패턴 검색으로 MZ·PE 시그니처 오프셋을 즉시 확인 가능
- `file.seek()` → `file.write()`로 오프셋 지정 편집 구현 — 패킹 파일의 EP 바이트 수정, 섹션 헤더 Characteristics 변경 등을 별도 도구 없이 수행
- 비PE 파일 로드 시에도 헥스 뷰어만은 정상 작동하도록 예외 분기를 구현

### QA & 디버깅

기능 통합 과정에서 발생한 버그를 발견·수정하고, 회귀 방지를 위한 검증 시나리오를 반복 수행했습니다.

| 버그 | 원인 | 해결 |
|------|------|------|
| 섹션 이름 빈 셀 표시 | NULL 바이트 포함된 섹션 이름을 `decode` 없이 처리 | `decode('utf-8', errors='replace')` 적용 |
| PDF 한글 깨짐 | 환경별 맑은 고딕 폰트 경로 불일치 | 폰트 파일을 프로젝트에 포함하여 경로 의존성 제거 |
| `PEFormatError` 앱 전체 종료 | 예외 미처리 | try-except 블록으로 감싸 헥스 뷰어는 계속 사용 가능하도록 예외 흐름 설계 |

정상 실행 파일, UPX 패킹 파일, 비PE 파일을 포함한 다양한 샘플로 통합 검증 시나리오를 작성하고, 기능 통합 시마다 반복 수행하여 회귀 버그를 방지했습니다.

## 기술 의사결정 및 트러블슈팅

### Problem 1. PE 포맷 오류 처리

**문제:** PE 형식이 아닌 파일(텍스트, 이미지 등)을 로드하면 `pefile.PEFormatError`가 발생하여 프로그램이 종료되었습니다.  
**해결:** 예외 처리로 경고 메시지를 표시하되, 헥스 뷰어로는 계속 접근 가능하도록 분기하여 모든 바이너리 파일을 분석할 수 있게 했습니다.

### Problem 2. 파편화된 도구 통합

**문제:** PEview, HxD, Detect It Easy, 보고서 도구를 각각 사용해야 하여 분석 흐름이 끊어졌습니다.  
**해결:** 단일 GUI에 전체 기능을 통합, 파일 로드 한 번으로 헤더 분석부터 PDF 보고서까지 원클릭으로 이어지는 워크플로우를 구현했습니다.

### Problem 3. 헥스 에디터 읽기 전용 한계

**문제:** 단순 헥스 뷰어는 읽기만 가능하여, 패킹 파일의 특정 바이트 수정 시 별도 에디터가 필요했습니다.  
**해결:** 오프셋 주소와 16진수 데이터를 입력하면 `file.seek()` → `file.write()`로 파일에 직접 반영되도록 편집 기능을 추가했습니다.

### Problem 4. 의존성 자동 관리

**문제:** PyQt5, pefile, reportlab 미설치 시 `ImportError`로 초보 사용자가 실행하지 못하는 문제가 있었습나다.  
**해결:** `main.py`에서 `__import__()`로 모듈 존재 여부를 확인하고, 없으면 `subprocess`로 pip 자동 설치를 수행하여 Python만 설치되어 있으면 바로 실행 가능하도록 했습니다.
